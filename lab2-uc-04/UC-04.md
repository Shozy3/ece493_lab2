## Use Case UC-04: Submit Manuscript

**Goal in Context**: An author submits a manuscript with required information to enter the review process.

**Scope**: Conference Management System

**Level**: User Goal

**Primary Actor**: Author

**Secondary Actors**: File Storage Service, Database, Validation Service

**Trigger**: User requests to submit a new manuscript.

### Success End Condition
* Manuscript is stored with all required metadata, file is uploaded successfully, and author receives confirmation with submission reference.

### Failed End Condition
* Manuscript is not submitted, no record is created, and author is informed of the failure.

### Preconditions
* User is registered and logged in as an author.
* Submission system is online and accessible.

### Main Success Scenario
1. User requests submission.
2. System redirects to submission page.
3. User fills required information (authors/affiliation/contact, abstract, keywords, main source) and uploads manuscript file (PDF, Word, or LaTeX format, max 7MB).
4. User clicks submission button.
5. System validates all fields and file rules (format, size).
6. System stores the submission in the database.
7. System shows success message and redirects user to home page.

### Extensions
* **5a**: Some fields invalid/blank/invalid email
  * 5a1: System informs user and shows error message indicating invalid fields.
  * 5a2: User corrects fields and resubmits.
* **5b**: File format not PDF/Word/LaTeX
  * 5b1: System informs user that file violates allowed formats.
  * 5b2: User uploads file in acceptable format and resubmits.
* **5c**: File size exceeds 7MB limit
  * 5c1: System displays error message indicating file is too large.
  * 5c2: User compresses file or reduces size and resubmits.

### Related Information
* **Priority**: High
* **Frequency**: Frequent during submission period
* **Open Issues**: None
