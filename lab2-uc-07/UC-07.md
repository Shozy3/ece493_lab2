## Use Case UC-07: Accept or Reject Review Invitation

**Goal in Context**: A referee accepts or rejects a review invitation to manage their review workload.

**Scope**: Conference Management System

**Level**: User Goal

**Primary Actor**: Referee

**Secondary Actors**: Email Service, Database, Editor

**Trigger**: Referee receives review invitation email and clicks accept or reject link.

### Success End Condition
* If accepted: Paper is placed under referee's account, referee can access paper for review, and editor is notified of acceptance.
* If rejected: Editor is notified of rejection, paper remains available for assignment to another referee.

### Failed End Condition
* Invitation response is not processed, and editor is not notified.

### Preconditions
* Referee received an invitation (from FR6 flow) and has an account.

### Main Success Scenario
1. Referee receives review invitation email.
2. Referee clicks accept link in email.
3. System verifies invitation is valid and not expired.
4. System checks if paper already has three assigned reviewers.
5. System places paper under referee's account.
6. System counts assigned reviewers for that paper.
7. System notifies editor that referee accepted invitation.

### Extensions
* **2a**: Referee clicks reject link in email
  * 2a1: System verifies invitation is valid and not expired.
  * 2a2: System records rejection in database.
  * 2a3: System notifies editor that referee rejected invitation.
  * 2a4: System redirects referee to confirmation page.
* **4a**: Paper already has three assigned reviewers
  * 4a1: System displays message indicating paper is fully assigned.
  * 4a2: System does not add paper to referee's account, editor is notified.
* **5a**: Assigned reviewers exceed three after acceptance
  * 5a1: System informs the editor of the exceedance.
  * 5a2: System still adds paper to referee's account.

### Related Information
* **Priority**: High
* **Frequency**: Frequent during review assignment period
* **Open Issues**: None
