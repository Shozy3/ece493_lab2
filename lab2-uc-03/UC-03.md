## Use Case UC-03: Change Password

**Goal in Context**: A user changes their password to maintain account security.

**Scope**: Conference Management System

**Level**: User Goal

**Primary Actor**: Registered User

**Secondary Actors**: Database, Password Validation Service

**Trigger**: User requests to change password.

### Success End Condition
* Password is updated in database, user is notified of successful change, and new password is active.

### Failed End Condition
* Password remains unchanged, user is notified of failure reason.

### Preconditions
* User has an account and is logged in.
* User knows their current password.

### Main Success Scenario
1. User requests to change password.
2. System displays password change form.
3. User enters current password and new password.
4. System validates new password meets security standards.
5. System updates password in the database.
6. System confirms password change to user.

### Extensions
* **3a**: Current password incorrect
  * 3a1: System displays error message indicating incorrect current password.
  * 3a2: User retries with correct current password.
* **4a**: New password does not meet security standards
  * 4a1: System displays error message indicating password requirements.
  * 4a2: User enters compliant password and resubmits.

### Related Information
* **Priority**: High
* **Frequency**: Occasional
